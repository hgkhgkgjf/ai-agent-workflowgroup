# 危险信号检测（Red Flags）

主会话必须持续监测以下危险信号。检测到任何一个时，**立即阻止并采取对应行动**。

**适用范围**：
- **始终生效**：信号 #3、#5、#7、#8、#9、#10、#11 —— 与运行模式无关
- **仅工作流模式生效**：信号 #1、#2、#4、#6 —— 默认模式下用户未启动管道，这些不触发

## 信号清单

| # | 危险信号 | 行动 | 严重度 | 适用 |
|---|---------|------|--------|------|
| 1 | 需求不清就开始编码 | 停下来，建议用户 `/workflow-start` 启动 brainstorming | 🔴 高 | 工作流模式 |
| 2 | 没有实现计划就开始开发 | 停下来，先写 writing-plans | 🔴 高 | 工作流模式 |
| 3 | 实现 agent 说"搞定了"但没有验证证据 | 要求运行测试并展示输出 | 🔴 高 | 始终 |
| 4 | code-reviewer 审查只做了一个阶段 | 要求完成两阶段审查 | 🟡 中 | 工作流模式 |
| 5 | 审查发现问题但直接跳过 | 要求对应实现 agent 修复后重新审查 | 🔴 高 | 始终 |
| 6 | 工作流中有人想跳过关键 phase | 说清楚跳过的理由，让用户确认（phase 按需裁剪，但不能无声跳过） | 🔴 高 | 工作流模式 |
| 7 | 使用"应该没问题""看起来对了" | 要求提供验证命令和输出证据 | 🟡 中 | 始终 |
| 8 | 同一类错误反复出现 | 运行 `bash scripts/harness/logs-query.sh --hotspots --days 30` 查看高频失败项 | 🟡 中 | 始终 |
| 9 | 文档与代码不一致 | 触发 [熵管理](../skills/workflow/entropy-management/SKILL.md)，修复漂移 | 🟡 中 | 始终 |
| 10 | Harness 传感器长期无告警 | 审查传感器有效性，可能检测不足 | 🟡 中 | 始终 |
| 11 | 主会话直接写代码/做设计/做测试 | 立即停下，按任务类型派遣专项 agent 执行 | 🔴 高 | 始终 |

## 升级机制

- 🔴 高：立即中断当前流程，回到正确阶段
- 🟡 中：在当前阶段结束前处理
- 信号 #8 的根本原则：**重复问题不应靠人工反复修正，而应编码为自动化约束**
